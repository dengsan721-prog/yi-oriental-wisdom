# 艺｜姓名使用与五行文化实施计划

目标：在不改变主盘计算、不增加出生录入负担的前提下，为已填写姓名的用户提供可追溯、可解释、能落到生活场景的姓名五行文化分析与保守改名建议。

## Task 1：建立来源、字形与离线数据契约

涉及文件：

- `site/lib/yi/name-data.ts`
- `site/lib/yi/name-types.ts`
- `site/lib/yi/sources.ts`
- `site/tests/yi/name-data.test.ts`
- `site/tests/yi/source-audit.test.ts`
- `site/THIRD_PARTY_NOTICES.md`

步骤：

1. 先写失败测试：官方字表码点集合与 8105 个 `kTGH` 记录无缺无增、原始 `2013:index` 和三级边界完整；常见字保留码点、读音、原始多值部首和单一 IRG 工程总笔画；不存在的字返回未知而非猜测。
2. 从 Unihan 17.0 裁出静态核心数据；读音优先 `kTGHZ2013`、回退 `kMandarin`；保存源 checksum、Unicode License v3、数据版本、生成日期和 URL。
3. 裁出 `kTraditionalVariant`／`kSimplifiedVariant` 候选索引但标注 provisional；禁止 NFKC 和静默换写。先按 raw grapheme/code points 检测兼容汉字、IVS 和 PUA，再决定是否生成 NFC 查表形式。依据《通用规范汉字表》附表建立首版常见一简多繁人工消歧表。
4. 增加失败样本：发／髮／發、后／後、台／臺／檯／颱、干／乾／幹、里／裏／裡，以及一个 NFC 前后码点会变化的真实兼容汉字、扩展区代理对、IVS、PUA。断言兼容字不被静默采用；按 grapheme cluster 处理，不按 UTF-16 code unit 拆字。
5. 建立逐字人工审校集和有限整名组合库。整名记录必须带具体姓氏、读音、字形口径、组合审校状态、日期、复核角色和风险；审过单字不得冒充审过整名。
6. 增加国家规范、原典、Unicode、简繁边界、五格历史与公安部姓名报告来源。每个 source ID 记录用途、不能支持的结论、版本／修订和访问日期；五行产品规则与国家规范使用不同 ID。
7. 8105 字数据做独立静态 chunk，压缩后预算 160 kB；空姓名不加载，运行时不请求第三方姓名接口。
8. 验证：focused tests、source audit、chunk budget、`git diff --check`。

## Task 2：实现姓名分析、透明评分与建议门禁

涉及文件：

- `site/lib/yi/name-analysis.ts`
- `site/lib/yi/name-score-contract.ts`
- `site/lib/yi/types.ts`
- `site/lib/yi/content-audit.ts`
- `site/tests/yi/name-analysis.test.ts`
- `site/tests/yi/content-audit.test.ts`

步骤：

1. 先写失败测试：空姓名返回 `null`；姓名变化不改变四柱或专业报告；分析器不回写主盘；多音、未知、争议保留；一简多繁未确认不静默换写；强制资料缺失不显示总分。
2. 实现 `rawInput` 保存、NFC 查表、grapheme 切分、常见复姓识别和逐字资料合成。主分析采用现实字形；传统字形必须用户主动确认，展示采用字形和笔画口径。
3. 删除部首五行权重。语义五行只读取带 `methodId`、版本、依据和争议状态的人工审校记录；五元素向量归一化，`unknownShare` 不作为第六元素。
4. 定义只读 `NameChartInteractionInput`，复用 `PillarKey`（`hour`）和结构化 `Pillar`：未知时辰只排除时柱，年／月待核时并读 unavailable；首版只并排展示，不生成匹配、补益、生克或适配结论。
5. 将资料覆盖与分数分开。按独立版本化契约实现展开层可选的“30 秒现实使用实测”：听见读准 30、输入显示 25、证件系统 25、含义接受度 20。任一项未验证则无总分；资料收录、读音记录和进入审校库均不加分。
6. 增加 golden cases：全项通过 100、一个听读场景失败 85、任一未验证无总分、同一答案在姓名进入审校库前后分数不变。
7. 拆分 `AnalysisBlocker` 与 `ConfirmedUsageRisk`。资料或采用口径不足只能阻断分析，不能触发改名；微调只由一个已人工且用户确认的现实硬风险触发，重构只由两个以上现实硬风险或用户主动候选模式触发。书写复杂、三级字、流行度低只作软提示。
8. 首版输出“3 个命名方向＋审校示例字”。完整候选仅来自有限整名库；其他动态组合标“待人工复核”。公安部 2021 数据不得作为质量或性别推荐依据。
9. 将姓名模块全部可见字段、建议、候选、边界和 source IDs 接入内容审计；禁词增加“改名改命、最吉、必选、补足五行、康熙古法、公安保证批准”。
10. 验证：代表姓名、未知字、多音字、简繁无歧义、一简多繁、兼容字／IVS／PUA、未知时辰、年／月待核、同盘改名不变性、敏感性测试和禁用词扫描。

## Task 3：接入命盘的低脑力阅读界面

涉及文件：

- `site/components/yi/NameAnalysisSection.tsx`
- `site/components/yi/ChartSection.tsx`
- `site/components/yi/ResultShell.tsx`
- `site/app/globals.css`
- `site/tests/yi/name-analysis-view.test.ts`
- `site/tests/yi/chart-view-model.test.ts`
- 浏览器交互测试文件（按项目现有结构放置）

步骤：

1. 先写失败测试：姓名为空无 DOM 且不请求 chunk；姓名非空时位于出生事实之后、人生概览之前；第一层只有当前姓名、现实使用结论、一项行动和核验状态。
2. 展开层包含资料覆盖、可选“30 秒现实使用实测”及分数／未验证状态、四维规则、逐字资料、简繁采用口径、命局文化并读、建议、来源和边界。
3. 一简多繁使用生活语义问题；候选不默认勾选，未确认不阻塞其余阅读。姓名、候选、读音和字形状态不写入 URL；候选确认默认不持久化。
4. 将 `name` 传给 `ChartSection`，但保持主盘结果对象不含姓名推导字段。官方同名查询外链不预填姓名，离站前提示实名认证与后续数据处理边界。
5. 实现 390px 单列、无横向滚动、44px 交互、长姓名／扩展字形换行、`details/summary` 键盘可用和正确焦点。
6. 浏览器交互覆盖：读音确认、字形确认、切换候选后返回现用姓名、重新提交不同姓名不残留旧选择、键盘和焦点、空姓名不加载模块。
7. 验证：focused/full tests、content/source audit、lint、严格 TypeScript、GitHub build gate、chunk budget。

## Task 4：独立审查与浏览器验收

1. 新审查员核对来源边界、字形口径、评分规则、姓名不改主盘、建议门禁、隐私和移动端；Critical／Important 必须关闭后才通过。
2. 390×844 实测：有名／无名、资料完整／不足、多音待确认、简繁无歧义／一简多繁、扩展字形、候选方向、官方外链提示、无横向溢出。
3. 桌面实测：层级、密度、逐字记录和规则可读性；检查黑金视觉是否保持主次。
4. 对姓名数据 chunk 做真实网络请求观察；确认无姓名时不加载、无第三方接口。
5. 运行完整门禁；只在证据齐全后进入后续四层报告和公开语言清理任务。

成功标准：

- 姓名为空时零展示、零额外操作、零姓名数据请求。
- 有姓名时 5 秒内看懂“现实中是否好用、名字表达什么、与命局怎样并读、是否值得改”。
- 现用字不被静默换写；传统字形、采用字形和笔画口径可见，一简多繁未确认时拒绝相关确定结论。
- 资料存在不加分；评分只来自用户自愿完成的现实使用实测且每项有规则 ID，任一项未验证就不显示总分，命局文化互动不计分。
- 不出现“缺什么补什么”“改名改命”“五格是中国古法”“公安一定批准”等表述。
- 现用姓名不会仅因分数或文化五行互动而被建议更改。
- 所有新增来源都明确“支持什么／不能支持什么”。
